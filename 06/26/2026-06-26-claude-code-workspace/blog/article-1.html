构建一个新闻聚合器：从 RSS 到分布式爬虫

PythonArchitectureCrawler

2026 年初，我决定从零开始构建一个覆盖 40+ 信息源的新闻聚合系统。最初只是想把晨间浏览新闻这件事自动化，但很快就发现这件事远比想象中复杂。这篇文章记录整个构建过程、踩过的坑以及最终的架构选择。

第一版：纯 RSS

最初的方案很简单——Python 标准库加上 RSS 解析就够了。feedparser 加上 asyncio，半小时就能跑通一个原型。每天定时拉取几个 RSS 源，去重后推送到飞书，轻松愉快。

问题很快就暴露了：高价值的信息源——微博热搜、GitHub Trending、华尔街见闻——根本不提供 RSS。它们的内容只能通过网页抓取获得。于是第二版的爬虫引擎诞生了。

第二版：加入 Web Scraping

引入 requests + beautifulsoup4 后，信息源从 15 个扩展到 40 个。但这带来了一整套新的工程问题：

首先是反爬对抗。不同网站的反爬策略天差地别——有的检查 User-Agent，有的做 TLS 指纹检测，有的在响应里埋 honeypot 链接。我开始维护一个源级别的配置表，为每个源定制请求头、延时策略和重试逻辑。

其次是内容去重。不同源经常报道同一条新闻，标题相似但措辞不同。简单的 SimHash 效果不好，最终用了 MinHash + LSH 的组合方案，准确率提升到 93% 左右。

最终架构

在迭代了四个版本之后，架构稳定下来：

news.py (CLI 入口)
  ├── sources/          # 40+ 信息源适配器
  │   ├── rss/          # RSS 源 (feedparser)
  │   ├── web/          # Web 抓取 (requests + bs4)
  │   └── api/          # API 源 (GitHub, 微博)
  ├── dedup/            # 去重引擎 (MinHash + LSH)
  ├── translate/        # 可选翻译模块
  └── notify/           # 推送模块 (飞书/钉钉/邮件)

每个源都是一个实现了统一接口的适配器，需要时可以通过 --source 参数单独调用，也支持分类聚合查询。

经验教训

最大的教训是：不要过早做分布式。第三版我曾试图用 Celery + Redis 做分布式爬虫架构，结果调度开销比爬取本身还大。回退到单进程 asyncio 之后，40 个源 8 秒内就能全部跑完。

第二个教训是：RSS 源的生命周期比想象中短。15 个初始 RSS 源中，不到一年就有 3 个停止更新、2 个更换域名。需要持续监控和维护。

目前这个聚合器已经稳定运行了三个月，每天早上自动推送，出错率不到 2%。如果你也在做类似的事情，可以直接用我的脚本——不需要 API Key，纯 Python 实现。
