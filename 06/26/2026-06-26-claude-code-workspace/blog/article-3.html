Claude Code 自动化工作流搭建笔记

AIAutomationDevOps

2026 年春天，我开始尝试将 Claude Code 融入日常工作流。动机很朴素——每天早上的新闻简报、每次 push 前的代码审查、以及周末的自动部署，这些重复劳动如果能让 AI 代劳，省下来的时间可以去做更有创造力的事情。一个月下来，我把三条主要管线跑通了，记录一下心得。

每日新闻推送管线

最早跑通的是新闻简报。每天早上 8 点，GitHub Actions 触发一个 Python 脚本，从 40+ 个信息源拉取当日科技新闻，去重排序后送给 Claude，由它生成一段 200 字左右的摘要。整个过程从触发到飞书卡片推送到手机，大约 3 分钟。

关键配置其实很简单——在 .github/workflows 下新建一个 workflow 文件即可：

name: Daily News Digest
on:
  schedule:
    - cron: '0 0 * * *'   # UTC 00:00 = 北京时间 08:00
  workflow_dispatch:       # 支持手动触发
jobs:
  digest:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: { python-version: '3.12' }
      - run: pip install -r requirements.txt
      - run: python scripts/news_daily.py
        env:
          FEISHU_WEBHOOK: ${{ secrets.FEISHU_WEBHOOK }}

有几个踩坑值得分享。第一，cron 时区陷阱——GitHub Actions 的 schedule 使用 UTC，所以北京时间 8 点要写成 0 0 * * *。第二，飞书 Webhook 的签名校验不是必选项，但如果你开启了，需要额外计算一个 HMAC-SHA256 签名追加到请求头里，否则消息会被静默丢弃，没有任何报错提示。第三，workflow 长时间不运行的仓库会被 GitHub 自动停用 schedule——需要每 60 天内至少有一次提交才能保持活跃。

代码审查自动化

这部分的灵感来自于一次 code review 马拉松。团队规模小，没有专门的 reviewers，每个人写完 PR 后经常要等半天才有人看。于是我试着让 Claude Code 做第一轮 review。

具体做法是：在 .github/workflows 下配置一个 PR trigger，当有人提交 PR 时自动运行 review。Claude 关注三类问题：

类型安全——Python 项目虽然没有编译期检查，但 Claude 能识别出潜在的类型不匹配。比如某个函数签名改了参数名但调用方没跟着改，这种问题人工 review 很容易遗漏。

复杂度红线——如果某个函数的圈复杂度超过了 10，或者一个 PR 改动了超过 500 行却没有拆分 commit，Claude 会在 review 中标注出来。这个规则很简单但效果出奇好。

安全陷阱——硬编码的密钥、未过滤的 SQL 拼接、debug 模式下的错误信息泄露。这些东西 Claude 非常敏感，几乎没有漏报。

不过需要明确的是，Claude 的 review 是辅助性的——我们仍然要求至少有一个人类 reviewer 批准才能合并。AI review 的作用是让人类 reviewer 的注意力更集中，不必在低级问题上浪费时间。

部署管线：GitHub Actions 到服务器

部署管线比前两条复杂一些，因为它涉及到 SSH 到远端服务器。我花了不少时间在安全配置上，最后的方案是这样：

name: Deploy
on:
  push:
    branches: [main]
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: pip install -r requirements.txt
      - run: pytest tests/
      - name: Deploy to server
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.SSH_HOST }}
          username: deployer
          key: ${{ secrets.SSH_PRIVATE_KEY }}
          script: |
            cd /opt/app
            git pull origin main
            docker compose up -d --build
            docker compose exec app python manage.py migrate

安全方面的几个要点：绝不要用密码登录——Ed25519 密钥对是当前最佳选择。为 CI 创建一个独立的服务器用户，权限严格控制到仅能执行部署操作。SSH 私钥存放在 GitHub Secrets 中，永远不要以任何形式出现在日志或配置文件中。

我还加了一个健康检查步骤——部署完成后 curl 一下生产环境的 /health 端点，如果 5 次重试后仍然返回非 200 状态码，则自动回滚到上一次的 Docker 镜像。这个自动回滚救过我两次。

经验与反思

一个月下来最大的感受是：AI 自动化真正的瓶颈不在模型能力，而在工程整合。Claude 本身很强大，但重要的是如何把它嵌到日常的 workflow 中，让它在你需要的时候自动出现，而不是需要你专门打开一个 Chat 窗口去问它问题。

另外一个小建议——给每一条自动化管线加一个 workflow_dispatch 触发器。不管你的 schedule 配得多完美，总有需要手动触发一次的情况。不加这个选项，到时候就得去 dev 分支上推送一个无关的空 commit 来触发 workflow，非常狼狈。

目前三条管线跑了一个月，稳定性不错。接下来打算加一条代码质量趋势追踪——每周自动跑一次 radon 和 pylint，生成趋势图贴到飞书群。如果有意思的话再写一篇后续。
