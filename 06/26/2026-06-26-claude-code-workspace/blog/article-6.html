从零读懂 Asar 打包格式

Reverse EngineeringElectron

Electron 应用用 .asar 文件打包资源这件事，几乎每个用过 Electron 的人都知道。但当我需要手动解析一个损坏的 asar 文件时，我发现自己对它的内部格式一无所知——只知道它是一个"类似 tar 的归档格式"。这篇文章记录了我从二进制层面拆解 Asar 格式的过程，以及为什么理解底层格式对日常开发也有帮助。

Asar 格式概览

Asar 不是压缩格式，它只是一个归档格式。这一点很重要——很多人以为 asar 像 zip 一样压缩了文件，但它实际上只是把所有文件拼接在一起，前面加了一个序列化的元数据头。这意味着 asar 内的文件可以随机访问，不需要先解压整个包。

整个 asar 文件由三部分组成：

  1. Header Size——一个 4 字节的无符号整数（小端序），表示后面序列化元数据的字节长度
  2. Header JSON——一个 Pickle 风格的 JSON 对象，描述虚拟文件系统树和每个文件在归档中的偏移量与大小
  3. File Contents——所有文件的内容按顺序拼接到一起。偏移量 0 指的是文件内容区域的起始位置
+------------------+-------------------+------------------------+
| Header Size (4B) | Header JSON (变长)  | File Contents (变长)    |
| uint32 LE        | UTF-8 JSON string  | raw bytes               |
+------------------+-------------------+------------------------+

注意两个细节：offset 字段在 JSON 中是字符串而非数字——这是 asar 规范的一个历史遗留，需要手动转成 int。另外，文件夹节点用嵌套的 "files" 键表示，形成了一个虚拟的文件系统树，这个结构可以直接还原为磁盘目录。



提取文件：从偏移量到磁盘



有了 header 中的 offset 和 size，提取单个文件就很简单了。关键是把 header JSON 中的偏移量加上内容区域的起始偏移（4 + header_size），然后直接 seek 过去读：



def extract_file(asar_path, target_path, output_path):
    header, content_offset = read_asar_header(asar_path)

    # 在 header 的 files 树中查找目标文件
    node = header['files']
    for part in target_path.split('/'):
        node = node[part] if part else node
        if 'files' in node:
            node = node['files']

    offset = int(node['offset'])  # string -> int
    size = node['size']

    with open(asar_path, 'rb') as f:
        f.seek(content_offset + offset)
        data = f.read(size)

    with open(output_path, 'wb') as out:
        out.write(data)



如果需要提取整个归档，只需要递归遍历 files 树，对每个文件节点调用类似的读取逻辑即可。30 行 Python 代码就足够实现一个最小化的 asar 解包工具——不需要 @electron/asar 也不需要 asar npm 包。



为什么理解底层格式很重要



你可能会说——我直接用 npx asar extract app.asar ./output 就行了，为什么要自己解析？在 95% 的情况下确实不需要。但剩下 5% 的情况，理解底层格式可以救你一命：



损坏的 asar 文件恢复——如果 Header Size 字段损坏了（比如 app.asar 下载不完整），官方的 asar 工具会直接报错拒绝解包。但如果你知道 header 的结构，可以手动猜测 header JSON 的大小，或者暴力搜索 JSON 的结束标志 }，然后跳过损坏的 header 直接读取文件内容区。



增量更新和差分分析——Electron 应用更新时，如果只想替换一个 JS 文件，不需要重新解包和打包整个 asar。理解了 offset 机制后，你可以直接修改目标文件的 offset 指向新内容。这在做应用热修复或 A/B 测试分发时非常有用。



逆向工程与安全审计——很多 Electron 应用把关键逻辑放在 asar 里，包括 API 密钥、加密逻辑、甚至硬编码的凭证。如果你在做安全审计（或仅仅是确认一个应用是否在你的机器上做了不该做的事），直接解包 asar 是最快的路径。



最后，自己动手解析一个二进制格式，是一种很踏实的理解方式。Asar 的格式足够简单——只有一个 4 字节头和一段 JSON——让你可以在一个小时内从零实现完整的读写能力。这种 "从 raw bytes 到结构化数据" 的体验，是单纯调用库函数无法替代的。如果你对二进制格式有兴趣，Asar 是一个极佳的入门练习。
