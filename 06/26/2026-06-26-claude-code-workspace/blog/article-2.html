为什么我选择 SQLite 作为独立开发的首选数据库

DatabaseSQLiteSide Project

PostgreSQL 很棒。MySQL 也很棒。但对于 side project 和独立开发来说，它们大多数时候都是过度设计。这篇文章聊聊为什么我的所有个人项目都首选 SQLite。

零运维是杀手特性

SQLite 不需要安装服务、不需要配置用户权限、不需要管理连接池。数据库就是一个文件。备份？cp data.db data.db.bak。迁移到新服务器？scp 一个文件就完事。

对于一个 side project 来说，运维开销是杀死项目的第一大原因。当你在下班后只有两个小时写代码时，你不想花半小时在调试 PostgreSQL 的认证问题上。

WAL 模式改变了游戏规则

SQLite 3.7 引入的 WAL（Write-Ahead Logging）模式让并发读写性能有了质的飞跃。在 WAL 模式下，读操作不会阻塞写操作，反之亦然。对于大多数 web 应用来说，这已经足够了。

PRAGMA journal_mode=WAL;
PRAGMA synchronous=NORMAL;
PRAGMA busy_timeout=5000;

这三行配置打开之后，SQLite 可以轻松支撑每秒数百次读写。对个人项目的流量来说，绰绰有余。

Litestream：实时备份到 S3

SQLite 最被诟病的一点是"不适合生产环境"。但 Litestream 的出现在很大程度上解决了这个问题——它实时流式复制 SQLite 的 WAL 日志到 S3 兼容存储，实现秒级 RPO 的灾难恢复。

配置只需要一行命令：

litestream replicate data.db s3://my-bucket/data.db

跑起来之后，你的数据库每一秒都在被备份到云端。服务器挂了，拿回文件就行。

什么时候不该用 SQLite

公平地说，SQLite 不适合所有场景：需要多机写入的分布式系统、超大并发（>1000 QPS 写入）、以及需要行级权限控制的场景，还是用 PostgreSQL 更好。但说实话，大部分 side project 永远达不到这个量级。
